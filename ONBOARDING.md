# RoomieFit — Ownership Handoff

This project has **no traditional backend and no user accounts** for the ~30
classmates using it — but *you* (the new owner) need access to two accounts
to keep it running. This doc lists exactly what those are and how to take
over each one. See `README.md` for what the app actually does.

Live app: https://valerion-exe.github.io/dorm_mates/

## The two accounts that matter

### 1. GitHub — repo `Valerion-exe/dorm_mates`
This is the code, the GitHub Pages hosting, **and** the shared data store
(`data/users.json` — every profile save is a commit to this file).

**To take over:**
- Simplest: get added as a **collaborator** (repo → Settings → Collaborators
  and teams → Add people). This is enough to push code and manage the repo.
- Full ownership transfer: repo → Settings → General → scroll to "Danger
  Zone" → **Transfer ownership**. This moves the repo to your GitHub account
  (or a new org). ⚠️ If the repo moves to a different owner/name, the Pages
  URL changes (`https://<new-owner>.github.io/<new-repo>/`), which breaks two
  things until updated — see "If the repo moves" below.

### 2. Cloudflare — hosts the Worker proxy
This is the account authorized via "Sign in with GitHub" that owns the
`dorm-mates-proxy` Worker (subdomain `ajay-jagan2020.workers.dev`). The
Worker holds the **`GITHUB_TOKEN` secret** — a GitHub fine-grained Personal
Access Token, scoped to only this one repo with Contents: Read & write
permission — and is the only thing in the system that can write to
`data/users.json`.

**To take over, pick one:**
- **Get added to the existing Cloudflare account** (Cloudflare dashboard →
  Manage Account → Members → Invite). You'd then have access to the same
  Worker at the same URL — nothing in the app needs to change.
- **Deploy your own Worker under your own Cloudflare account** (cleaner for a
  full handoff — see "Redeploying the Worker" below). This changes the
  Worker's URL, so `src/lib/github.js`'s hardcoded `WORKER_URL` needs updating.

## Rotating the GitHub token (do this on any ownership change)

The token currently in the Worker's secret was generated by the outgoing
owner and should be treated as theirs. Whoever ends up controlling the repo
should replace it:

1. Generate a new one: `github.com/settings/personal-access-tokens/new`
   - Resource owner: whoever owns the repo now
   - Repository access: **Only select repositories** → `dorm_mates`
   - Permissions: **Contents → Read and write** (nothing else)
2. From the `worker/` folder, run:
   ```
   npx wrangler secret put GITHUB_TOKEN
   ```
   and paste the new token when prompted (it's read from stdin, never
   written to a file).
3. Revoke the old token from the previous owner's
   `github.com/settings/personal-access-tokens` page.

No redeploy needed — secrets apply to the Worker immediately (allow ~15–20
seconds to propagate).

## Redeploying the Worker under a new Cloudflare account

Only needed if the new owner wants their *own* Cloudflare account rather than
being added to the existing one.

```
cd worker
npx wrangler login              # opens a browser — "Sign in with GitHub" is fine
npx wrangler secret put GITHUB_TOKEN   # paste a fresh repo-scoped token
npx wrangler deploy
```

The first deploy under a fresh Cloudflare account will prompt you to
register a free `workers.dev` subdomain (a one-time, no-cost step — pick
any available name). The deploy output prints the final URL, e.g.:
```
https://dorm-mates-proxy.<your-subdomain>.workers.dev
```

Then update `src/lib/github.js`:
```js
const WORKER_URL = 'https://dorm-mates-proxy.<your-subdomain>.workers.dev/api/users';
```
Commit, push to `main` — the Pages deploy picks it up automatically.

## If the repo moves to a different GitHub owner/name

Two places hardcode `Valerion-exe`/`dorm_mates` and need updating, plus the
Pages source setting:

1. `worker/src/index.js` — the `REPO` constant (`owner`, `repo`, `branch`).
   Redeploy the Worker after changing it (`npx wrangler deploy` from `worker/`).
2. `worker/wrangler.toml` — `ALLOWED_ORIGIN` var, must match the new Pages
   URL exactly (`https://<new-owner>.github.io`), or the Worker will reject
   the app's requests via CORS.
3. `vite.config.js` — `base: '/dorm_mates/'` only needs to change if the repo
   is renamed (Pages project sites are served under `/<repo-name>/`).
4. New repo's **Settings → Pages → Source → GitHub Actions** must be set
   explicitly — it does not carry over automatically and defaults to
   "Deploy from a branch," which serves raw source and shows a blank page
   (this bit us once already — see git history around commit `2031308`,
   which removed a stray Jekyll workflow causing exactly this).

## Verifying the handoff worked

After rotating the token / redeploying, confirm both directions work without
touching the live roster:

```
curl -s "https://dorm-mates-proxy.<subdomain>.workers.dev/api/users" \
  -H "Origin: https://<pages-origin>"
```
Should return `{"data":[...],"sha":"..."}` with the current roster (17+
profiles as of this handoff). A `401`/`403` means the token or its
permissions are wrong; a CORS error in-browser means `ALLOWED_ORIGIN` doesn't
match the Pages origin.

Do **not** run a test write (PUT) against the live endpoint — it's real
classmates' data now, not a sandbox. If you want to confirm writes work,
do it through the actual app UI (edit your own profile and save) rather than
scripting a raw request.

## Everything else

Nothing else needs transferring. The ~30 classmates using the app have no
accounts, tokens, or logins of any kind — they just open the site and type
their name.
